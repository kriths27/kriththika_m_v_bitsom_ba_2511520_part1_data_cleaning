### 2.2 Discount Validation Rules
- Discounts outside [0, 1] were **flagged** as `INVALID_DISCOUNT`.
- `cleaned_discount` was computed by **clamping** discount to [0, 1] — negative values → 0, values > 1 → 1.
- Null discounts → default to 0 in `cleaned_discount`.

### 2.3 Date Standardization Rules
- Raw dates had **mixed formats** (e.g., `21 Jul 2024`, `06/08/2024`, `28-11-2024`, `2025-09-01`).
- Applied multi-format parser attempting 8 different format strings and a fallback `dayfirst=True` parser.
- Records with **unparseable dates** were flagged but retained (not dropped).
- Records where `ship_date < order_date` were flagged as `SHIP_BEFORE_ORDER`.

### 2.4 Duplicate Handling
- Detected 32 duplicate `order_id` entries (keeping first occurrence).
- Duplicates retained in raw data with flag `DUPLICATE`.
- **Excluded** from `cleaned_orders.xlsx` and all pivot analyses.

## 5. Assumptions & Limitations

1. **Discount Clamping:** Out-of-range discounts were clamped rather than dropped to preserve record count. The original discount value is retained in the `discount` column.
2. **Date Parsing:** Multiple date formats in source data suggest inconsistent data entry. Unparseable rows (if any) were flagged but not removed.
3. **Profit Calculation:** `calculated_profit = sales − cost` is used as the primary profit figure because `profit` in the raw file may have been pre-computed using a different rounding or formula.
4. **Duplicate Treatment:** First occurrence of a duplicate `order_id` is retained as authoritative. No attempt was made to merge/reconcile duplicate records.
5. **Returned Orders:** Treated as revenue reversals. Summarized separately to allow leadership to view gross vs. net revenue.
6. **Unknown Region/Ship Mode:** Filled with "Unknown" rather than dropped, to ensure order counts are not understated and downstream analysis can still flag these for follow-up.
